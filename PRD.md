# Vibe Learner PRD

## 1. 产品概述

Vibe Learner 是一个面向技术学习者的 AI 交互式学习产品。它把技术主题从“资料散落在文档、博客、论文和代码里”的状态，转成可学习、可练习、可复盘的课程体验。

第一阶段以 Ray 为样板主题，并加入 vLLM 种子主题，验证核心体验：

- Topic Hub 管理多个技术主题
- Agent 先调研主题
- 系统生成结构化课程
- 学习者按章节学习
- 通过互动练习检测掌握度
- 根据错题和回答形成复盘建议

长期目标是成为一个“AI 技术学校”：用户输入一个技术主题，系统自动构建课程、题库、讲解、追问和复习路径。

## 2. 背景与问题

技术学习常见痛点：

- 官方文档信息完整，但学习路径不清晰
- 博客和视频质量不稳定，知识点容易碎片化
- 学完概念后缺少练习和反馈
- 不知道自己到底哪里没掌握
- 框架版本变化快，学习材料容易过期
- AI 问答能解释问题，但很难持续组织成课程和训练

Vibe Learner 要解决的不是“回答一个问题”，而是“组织一段学习旅程”。

## 3. 目标用户

### 3.1 主要用户

技术学习者：

- 学习新框架的工程师
- 准备面试或转岗的开发者
- 想系统理解 AI Infra / Backend / Distributed Systems 的学习者

### 3.2 次要用户

内容生产者：

- 技术讲师
- 开源项目维护者
- 企业内部培训负责人

他们可以用 Vibe Learner 快速生成课程草稿、练习题和知识检查。

## 4. 核心用户场景

### 场景 A：学习一个新框架

用户输入 “Ray”，系统生成：

- 学习路径
- 架构讲解
- 核心概念卡片
- 代码示例
- 练习题
- 错题复盘

用户完成课程后，能解释 Ray 的 Task、Actor、ObjectRef、Object Store、调度器和资源声明。

### 场景 B：检验自己是否真的学会

用户读完某一章后进入练习，系统用选择题、代码阅读题、debug 题和场景题测试掌握度。

答错后，系统不是只给答案，而是指出对应薄弱知识点，并推荐回看章节。

### 场景 C：快速生成一个技术主题课程

用户输入 “vLLM”，Survey Agent 调研官方文档、论文和示例代码，生成课程草稿。用户可以审核、编辑、发布。

## 5. 产品定位

Vibe Learner 不是：

- 普通聊天机器人
- 文档搜索工具
- 单纯题库系统
- 视频课程平台

Vibe Learner 是：

- Agent 驱动的技术课程生成器
- 交互式学习和练习系统
- 带错题本和掌握度反馈的 AI Tutor
- 面向工程技术主题的学习操作台

## 6. 信息架构

### 6.1 当前 MVP

- 主题：Ray、vLLM seed
- Topic Hub：主题选择入口
- 讲解页：章节内容、概念卡片、架构图
- 练习页：题目、选项、提交、反馈
- 复盘页：错题本、下一步路线

### 6.2 目标结构

- Home：选择或创建学习主题
- Survey：查看 agent 调研结果和来源
- Course：章节式学习
- Practice：练习和考试
- Review：错题本、薄弱点、复习计划
- Builder：课程生成和人工审核

## 7. 核心功能需求

### 7.1 主题学习

用户可以选择一个技术主题并进入学习路径。

需求：

- 显示主题简介
- 显示推荐学习顺序
- 显示章节完成状态
- 支持继续上次学习进度

优先级：P0

### 7.2 章节讲解

每个章节应包含：

- 核心概念
- 一段结构化讲解
- 示例代码
- 架构图或流程图
- 常见误区

优先级：P0

### 7.3 互动练习

题型包括：

- 单选题
- 多选题
- 判断题
- 填空题
- 代码阅读题
- debug 题
- 场景设计题

提交后展示：

- 是否正确
- 正确答案
- 答案解析
- 关联知识点

优先级：P0

### 7.4 掌握度模型

系统需要根据答题记录估算掌握度。

第一阶段：

- 按答对题数计算总掌握度
- 按知识点统计正确率
- 记录错题

后续阶段：

- 根据题目难度加权
- 根据自由回答质量评分
- 根据遗忘曲线安排复习

优先级：P1

### 7.5 错题本

错题本应展示：

- 题目
- 用户答案
- 正确答案
- 解析
- 关联知识点
- 回到对应章节的入口

优先级：P0

### 7.6 Survey Agent

Survey Agent 负责调研技术主题。

输入：

- 主题名称
- 可选版本
- 可选学习目标

输出：

- 主题地图
- 资料来源
- 核心概念
- 学习路径
- 代码示例候选
- 常见误区
- 题库素材

优先级：P1

### 7.7 Course Builder Agent

Course Builder Agent 负责把 survey 结果整理成课程。

输出：

- 章节
- 概念卡片
- 示例代码
- 练习题
- 答案解析
- 复习建议

优先级：P1

### 7.8 Interactive Tutor

Interactive Tutor 负责追问和解释。

能力：

- 接受用户自由回答
- 判断回答是否正确
- 给提示而不是直接给答案
- 根据错因追问
- 把对话转成学习记录

优先级：P2

## 8. 非功能需求

### 8.1 内容可靠性

- 技术内容要尽量来自官方文档、论文或源码
- 每个自动生成主题应保存来源
- 对版本敏感内容要标注版本
- Agent 输出需要可人工审核

### 8.2 可扩展性

- 课程内容应数据化
- 前端不应硬编码具体主题
- 题库应支持多题型
- 学习记录应可持久化

### 8.3 易启动

本地开发应支持一键启动：

```bash
./scripts/start.sh
```

后续迁移框架后仍应保留统一启动入口。

## 10. 版本路线

### v0.1：Ray 静态原型

目标：验证学习体验。

包含：

- Ray 课程
- 练习题
- 错题本
- 一键启动
- README / PRD / 开发计划

### v0.2：数据化课程

目标：让内容可以由 agent 生成。

包含：

- `data/ray.json`
- `course.schema.json`
- 多题型题库
- 内容校验脚本
- 重置进度

### v0.3：多主题学习

目标：支持多个技术主题。

包含：

- 主题首页
- 主题切换
- 多课程数据
- 通用学习状态模型

### v0.4：Survey Agent 原型

目标：自动生成课程草稿。

包含：

- 输入主题
- 拉取资料
- 输出 survey report
- 生成课程草稿
- 人工审核

当前离线版本已包含：

- 结构化 survey report 数据模型
- Ray/vLLM 两个种子调研报告
- 离线 report 生成命令
- 页面内自定义主题输入入口
- report 校验与课程章节引用检查
- 复盘页展示调研摘要、概念地图、误区、题目种子和来源

### v0.5：Interactive Tutor

目标：从做题系统升级为 AI 老师。

包含：

- 自由问答
- 追问
- 提示
- 回答评分
- 个性化复习建议

## 11. 成功指标

### 产品指标

- 用户能在 5 分钟内理解一个主题的核心结构
- 用户完成一章后能通过对应练习
- 用户知道自己的薄弱知识点
- 用户愿意继续学习下一章

### 内容指标

- 每个章节至少 3 个核心概念
- 每个章节至少 5 道练习题
- 每道题都有答案和解析
- 自动生成内容可追溯来源

### 工程指标

- 一条命令可启动
- 课程数据可独立维护
- 新增主题不需要改核心 UI
- 关键交互有测试覆盖

## 12. 风险与应对

### 风险：AI 生成内容不准确

应对：

- 保存来源
- 优先官方资料
- 增加人工审核
- 增加内容校验

### 风险：产品变成普通问答

应对：

- 强化课程路径
- 强化练习和复盘
- 把对话结果沉淀为掌握度

### 风险：题目质量不稳定

应对：

- 建立题目 schema
- 区分题型和难度
- 增加参考答案和评分 rubrics

### 风险：主题扩展后 UI 混乱

应对：

- 尽早抽离数据模型
- 建立统一主题结构
- 先支持少量高质量主题

## 13. 开放问题

- 第一批主题应该选 AI Infra，还是更通用的后端技术？
- Survey Agent 是否需要联网实时调研，还是先用人工提供资料？
- 自由问答评分是否需要引入 rubrics？
- 学习进度是只保存在本地，还是要做账号体系？
- 课程生成后是否需要一个审核发布流程？
